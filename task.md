## Блок 1: Инициализация проекта

Мы берем [FastAPI-template](https://github.com/s3rius/FastAPI-template/tree/master) и инициализируем проект. Также разбиваем всё на эндпоинты, сервисы и инфраструктуру. Что нужно сделать? Нужен всего лишь один эндпоинт, в который будут передаваться параметры поиска по GitHub-репозиториям


## Блок 2: Работа с GitHub API

У GitHub репозиториев есть ручка `search`. В документации хорошо описано, как искать репозитории. Мы обращаемся к [`search repositories`](https://docs.github.com/en/rest/search/search?apiVersion=2022-11-28#search-repositories) и передаем запрос на поиск. Он выглядит следующим образом: основной текст запроса, язык, на котором написан код, сортировка и её порядок (order). DSL языка запросов, который использует GitHub, также можно посмотреть в документации.

## Блок 3: Требования к функционалу

Нужно сделать одну единственную ручку в FastAPI приложении. Она не будет обращаться к базе данных, а просто сохранять CSV-файл с найденными репозиториями. Формат файла должен совпадать с тем, что был в первой домашней работе. Ручка должна принимать количество репозиториев, которые мы ищем. Это нужно, чтобы выводились не все (например, 10 000 репозиториев по Python), а определенное количество, так как у ручки `search` есть пагинация (запрос данных по частям)

**Что принимает на вход ручка:**
- limit - сколько репозиториев нужно вернуть
- offset - если репозиториев нашлось больше, чем заданный лимит, то должно вернуться заданное количество со смещением на offset
- lang - язык программирования репозитория
- stars_min - минимальное количество звезд у репозитория (по-умолчанию 0)
- stars_max - максимальное количество звезд (по-умолчанию не ограничено)
- forks_min - минимальное количество форков у репозитория (по-умолчанию 0)
- forks_max - максимальное количество форков (по-умолчанию не ограничено)

После реализации и создания такого решения,  делайте MR в общий репозиторий. В mr каждая команда получит один дополнительный фильтр на реализацию
## Блок 4: Архитектура решения

При запросе ваш GitHub HTTP-клиент (как его реализовать, я показывал на одном из семинаров) идет в GitHub, формирует запрос с параметрами, получает данные и записывает их в CSV-файл по нужной структуре.  
В итоге должно быть реализовано:

1. Ручка (endpoint).
    
2. Клиент (в папке `infrastructure`).
    
3. Сервис, который формирует запрос и создает CSV-файл.
    

Результат работы — вызов ручки с определенными параметрами и появление в папке `static` CSV-файла с репозиториями по имени `repositories_{lang}_{limit}_{offset}.csv`.

## Блок 6: Сроки сдачи и критерии приёмки

В задании написано сделать до 15 декабря, но на самом деле делайте до 20 декабря. До 25-го числа мы всё проверим
Критерии:
- команда `make lint-check` проходит без ошибок для вашего сервиса
- формируется итоговый файл с заданным именем и структурой. В файле содержатся только репозитории, соответствующие фильтрам
- Структура проекта соответствует указанной в задании
- Используется [fastapi](https://github.com/fastapi/fastapi) как веб фреймворк, [descanso](https://github.com/reagento/descanso/tree/develop) или сырой [httpx](https://github.com/encode/httpx/tree/master/httpx), [aiofile](https://github.com/mosquito/aiofile) для работы с файловой системой
